Qu'est-ce que GIT ?

Outil de versioning

Similaire à :

  • SVN
  • CVS
  • Mercurial

Créé par Linus Torvalds (linux)

Git est un outil qui permet de versioner des fichiers. D'autres systèmes existent, on citera par exemple SVN qui était très répendu avant Git, mais aussi CVS ou Mercurial. Git a été créé par Linus Torvalds, qui est également connu pour avoir créé le noyau linux.

Utiliser git

Annuler une modification de fichier

git checkout ...

changer de branche

git checkout ...

créer une branch

git checkout ...
Git est difficile a appréhender pour les débutants face à d'autres système de versioning tel que SVN. Il y'a plus d'étapes pour effectuer des actions simples comme un commit poussé, plus de commandes similaires pour des résultats différents. Mais c'est aussi pour toutes ces raisons que Git est plus puissants puisqu'il permet plus de choses, découple mieux les étapes, et permet une meilleure organisation. Le but de cette formation est donc de présenter le fonctionnement et l'utilisation de Git, mais aussi de fournir quelques outils et bonnes pratiques afin d'augmenter l'efficacité de son utilisation.

Alors pourquoi GIT ?

  • Performances
  • Structure de développement
  • Meilleures gestion des conflits
  • Local repository
  • Gestion de l'historique
  • Actuellement le plus répandu (support et intégration)
Pourquoi utiliser git plutôt qu'un concurrent comme SVN qui peut paraître plus simple et plus rapide à utiliser ? - Performance : Git, de par sa gestion du versioning par snapshot (voir prochain chapitre) et par son système de dépôt local, est plus rapide. Comme tout se passe en local (status, diff, branch,...), nous sommes indépendants d'un serveur et ainsi, des possibles latences ou pertes de connexion au réseau. - Structure de développement : La création de branches est très rapide avec git, ce qui pousse à utiliser cette structure et ainsi améliorer l'organisation du développement du projet dans les Teams - Gestion des conflits : Git favorise les commit nombreux mais léger, ce qui permet de mieux découpler le versioning, mais également de faciliter les fusions. De plus, les nombreux témoignages disponibles sur Internet s'accordent à dire que les conflits sont mieux géré par git que par ses concurrents. (Ex: pas de tree conflict svn) - Local repository : Avoir un dépôt local, permet de transformer facilement n'importe quel projet en un projet versioné. Cela permet aussi de faciliter les échanges en poussant directement nos versions vers un dépôt distant X ou Y. - Gestion de l'historique : L'utilisation de branches et les commandes tel que rebase, merge fast-forward, etc. permettent d'avoir un historique clair et fortement découplé en de nombreux commit léger, sans toutefois devoir ajouter des commit de fusion. - Support et intégration : Git est actuellement l'outil de versioning le plus utilisé dans le monde Open Source et sur Internet. Pour cette raison il est actuellement le plus souvent intégré nativement ou sous forme de plugin dans les IDE ou autres applications de développeurs (Sublime Text, Visual Studio, Visual Studio Code, IntelliJ suite, etc.)

Snapshot de version (1/2)

Snapshot VS
versioning de fichier

Un snapshot dans le cas de git, c'est un état du système de fichier à l'instant T. Un snapshot est très léger et ainsi beaucoup plus efficient qu'un versioning de fichier, qui lui, recopie chaque fichier modifié.

Snapshot VS Version (2/2)

Gestion d'un système de fichier Liste de fichier
Snapshot la workspace Chaque modif génère une version de fichier
Fichier non modifié = lien ref précédente Fichier non modifié = pas de version

Git Flow

Master

  • Stable
  • Déployé en PROD
  • Pas de commit direct
  • Merge de release ou hotfix

Develop

  • Branche de développement
  • Source = master
  • Pas de commit direct
  • Merge de features, bugfixes et hotfixes

Feature branch

  • Nouvelle fonctionnalité
  • Source = develop | features branch
  • Target = parent branch
  • Commit direct
  • Merge de branch enfant et/ou parent

Release branch

  • Stabilisation de release
  • Plus de développement (bugfix excepté et a merger dans develop)
  • Target = master pour déploiement
  • Commit direct pour bugfix uniquement
  • Bugfix lourd = abandon de la branche

Hotfix

  • Bugfix sévère sur production
  • Source = master
  • Target = master + develop
  • Commit direct

Tag

  • Tag de version déployée
  • Source = master
  • Semantic versioning (semver.org)

  • Bisect → Tests → good|bad
  • Jeu : trouve un nombre entre x et y
git bisect <start | good | bad | reset>

Une fois poussé, ne plus toucher

D'une fois qu'un changement a été poussé sur le repository public, il ne faut plus le modifier (revert, patching,...). Une modification de l'historique peut rendre très difficile l'adaptation des autres publisher du projet. C'est également pour cette raison qu'un rebase ne devrait jamais être effectué sur une branche partagée.
